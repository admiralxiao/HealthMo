# 防止代码抄袭与逆向（提高成本的组合拳）

更新时间：2026-01-30  
目标：降低“被直接复刻”的概率，并让二开者付出明显成本。

---

## 1. 技术侧（提高逆向成本）
### 1.1 Swift/ObjC 代码加固（MVP→Level1）
- Release 构建：
  - 符号剥离（strip）
  - 关闭调试日志与 debug 菜单（生产）
- 结构设计：
  - 关键策略/参数通过远程配置下发（你已具备 docs/12）
  - 关键价值（Explain/趋势）尽量服务端化

### 1.2 API 保护（比混淆更重要）
- 所有核心价值都需要服务端签名响应
- 请求加入：
  - nonce + timestamp（防重放）
  - request_id（追踪）
  -（Level1）App Attest assertion（防伪造客户端）

### 1.3 “不可见水印”（建议）
- 在输出数据中加入轻量 watermark（不影响用户）：
  - 例如对 score 的小数部分编码版本/渠道（UI 显示整数）
  - 或对窗口推荐的排序加入可识别 pattern
- 目的：发现数据被抄用时可举证/追溯（不用于对用户展示）

> 注意：水印要足够隐蔽且不影响体验；不要把个人隐私编码进去。

---

## 2. 分发与运营侧（减少二开动机）
- 上线初期用 Phased Release + TestFlight 控制传播面
- 关键功能分阶段开放（灰度），让二开难以“立刻完整复刻”
- 快速迭代 UI/策略：让抄袭者永远落后一个版本

---

## 3. 法务与品牌侧（必要但不复杂）
- Terms/Privacy 明确：
  - 禁止逆向、复制、再分发
- 代码与品牌：
  - 商标注册（名称/Logo）
  - 关键 UI 视觉资产版权归属与授权清晰
- 发现二开：
  - DMCA/商店投诉路径准备（最有效的“停止传播”手段之一）

---

## 4. 可落地清单（MVP+）
- [ ] Release build 清理调试入口与日志
- [ ] 关键接口要求服务端授权（entitlement、explain）
- [ ] 远程配置与参数下发（已完成）用于快速调整策略
- [ ] 规划 watermark（可后置）
